# opencode

Configuration, agents, and skills for [opencode](https://opencode.ai).

## Contents

| Path | Description |
| --- | --- |
| `opencode.jsonc` | Main opencode configuration: providers/models, MCP server definitions, permission rules, and per-agent permission overrides. |
| `tui.json` | Terminal UI configuration (theme, etc.). |
| `skill-lock.json` | Lockfile tracking installed skills and their source repos, used by opencode's skill installer to manage updates. |
| `agents/` | Custom subagent definitions. |
| `skills/` | Agent skills (portable instructions invoked by name or automatically based on their description). |

### Agents (`agents/`)

| Agent | Purpose |
| --- | --- |
| `bun-code-reviewer.md` | Reviews JavaScript/TypeScript code targeting the Bun runtime (type safety, Biome compliance, Bun built-in APIs for SQL, RabbitMQ, S3, security). |
| `rust-code-reviewer.md` | Reviews Rust code, especially async services built on Tokio/Axum with PostgreSQL, RabbitMQ, or S3 integrations. |
| `spring-cloud-reviewer.md` | Reviews Java/Spring Cloud microservices for correctness, Spring best practices, resilience, security, and Kubernetes readiness. |
| `datadog-analyzer.md` | Reads and analyzes Datadog metrics, logs, traces, dashboards, monitors, and events to summarize system health or investigate incidents. |

### Skills (`skills/`)

| Skill | Purpose |
| --- | --- |
| `ax` | Use the `ax` CLI instead of ad hoc curl/scraping scripts to fetch and extract structured data from web pages. |
| `brainstorm` | Analyze an improvement ticket or idea and produce a detailed implementation plan. |
| `code-review` | Orchestrate language-specific reviewer subagents (e.g. `bun-code-reviewer`, `rust-code-reviewer`) to review recently written or modified code. |
| `datadog-health-report` | Produce a consolidated, meeting-ready Datadog health report by orchestrating sub-agents across metrics, logs, traces, monitors, SLOs, and incidents. |
| `grill-me` / `grilling` | Relentlessly interview the user to stress-test a plan or design before building it. |
| `handoff` | Compact the current conversation into a handoff document for another agent session to pick up. |
| `implement` | Execute an implementation plan previously generated by the `brainstorm` or `troubleshoot` skills. |
| `pretty-mermaid` | Render beautiful Mermaid diagrams as SVG or ASCII art with theming support. |
| `prototype` | Build a throwaway prototype to sanity-check a state model, logic, or UI design question. |
| `query-postgres` | Query Postgres databases via `psql` using per-environment credential files, defaulting to read-only. |
| `troubleshoot` | Analyze a problem or issue and produce a detailed troubleshooting plan. |

## Installation

opencode looks for skills and agents in different places, so each part of this folder is installed differently:

1. **Skills** are agent-agnostic and shared across tools, so they are symlinked into `~/.agents/skills/`:

   ```sh
   ln -s "$(pwd)/opencode/skills" ~/.agents/skills
   ```

   Or, if you'd rather link each skill individually (e.g. to mix in skills from elsewhere):

   ```sh
   for skill in opencode/skills/*/; do
     ln -s "$(pwd)/$skill" ~/.agents/skills/"$(basename "$skill")"
   done
   ```

2. **Agents and opencode-specific config** are tool-specific, so they are symlinked into your opencode config folder (`~/.config/opencode/`):

   ```sh
   ln -s "$(pwd)/opencode/agents" ~/.config/opencode/agents
   ln -s "$(pwd)/opencode/opencode.jsonc" ~/.config/opencode/opencode.jsonc
   ln -s "$(pwd)/opencode/tui.json" ~/.config/opencode/tui.json
   ```

The same pattern applies to other AI coding agents: link the `skills/` folder (or its individual skill subfolders) into `~/.agents/skills/`, and link any agent/tool-specific files into that agent's own config folder (e.g. `~/.copilot`, `~/.claude`, `~/.codex`, etc.), adapting paths and file names to what the target tool expects.

> [!WARNING]
> `opencode.jsonc` may contain embedded API tokens/secrets for MCP servers (Atlassian, GitHub, etc.). Treat it as sensitive, keep this repository private, or don't commit the tokens to version control.
